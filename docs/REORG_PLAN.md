`clawchat-agent-plugin/hermes-clawchat/`. It captures the audit findings,
the proposed target structure, and the commit-by-commit migration.

> Generated by the workspace docs reorg pass on 2026-05-27. Branch

> workspace paths so the audit is reproducible; the resulting docs use

## 2. Findings (grouped)

### 2.1 Structural clarity

- `README.md` is the project's only public-facing doc and tries to be
  everything: marketing intro, host install, Docker install, full 22-tool
  catalog (~20 lines), quickstart shell snippets, deep "Install With Hermes
  Plugins" explainer, prompt-injection mechanics, group-mode env-var
  reference. Three distinct audiences (operator / agent author / plugin
  developer) are interleaved.
- `install.md` is a duplicate operator runbook with subtly different facts
  (tool count) and a slightly different command set (no `hermes gateway
  setup` mention, no Docker block).
- `skills/clawchat/SKILL.md` is a runtime artefact — Hermes loads it as a
  bundled skill via `ctx.register_skill(...)` — but it doubles as the only
  agent-author documentation. There is no signpost from `README.md` to it.
- No root index, no audience map, no per-subdir README.
- No `AGENTS.md` / `CLAUDE.md` for in-tree assistants (the repo is tracked
  by `.gitignore` to exclude these anyway — see §5).

### 2.3 Duplication / errors / omissions

**Duplicated content (with proposed canonical home):**

- D1. 22-tool catalog: lives in `plugin.yaml` (`provides_tools`), in `README.md` lines 26-46 (descriptions), in `plugin_tools.py` (tool schemas), and partly in `SKILL.md` (routing table). Canonical: `plugin.yaml` for the list; `docs/reference/tools.md` for human descriptions; everything else links to those.
- D2. Install/activation commands repeated in `README.md` (top + "Install With Hermes Plugins" section), `install.md`, `SKILL.md` (table). Canonical: `docs/install.md` (operator runbook); README links once.
- D3. Docker command block repeated at `README.md:10-15` and `README.md:107-112`. Canonical: a single Docker section in `docs/install.md`.
- D4. "Hermes v0.12.0+ requirement" sentence appears 4× across `README.md` and `install.md`. Canonical: a single "Compatibility" line in `docs/install.md`.
- D5. Activation-code disclaimer ("single-use; ask for a fresh one on failure") appears in `install.md:52` and `SKILL.md:44`. Keep in both because they target different runtimes, but make `install.md` the canonical source.

**Errors / wrong claims:**

- E1. `install.md:32` "twenty-one" tools — fix to 22.
- E2. `README.md:103-105` claims `prompts/group.md` and `prompts/user.md` are injected — they are *optional* and currently absent. Rephrase as "if present" or list the actual files that ship.
- E3. `README.md:34` claims "after success, return exactly `""` and do not send a normal follow-up reply" — that wording is misleading: the **adapter** suppresses the same-turn follow-up reply (see `terminal_send.py` + `mention_message.py`), the tool *result* is a JSON envelope. The SKILL.md wording (line 58) is more accurate. Rewrite using the SKILL.md phrasing.
- E4. Nothing tells the reader that `clawchat-gateway` (PyPI/package name) ≠ `clawchat` (Hermes plugin identifier) ≠ `clawling/hermes-clawchat` (GitHub source spec). See C6.

**Omissions a new contributor would need:**

- O1. **Naming map**: Hermes plugin name (`clawchat`), Python package (`clawchat-gateway`), GitHub source (`clawling/hermes-clawchat`), skill qualified name (`clawchat:clawchat`), plugin install dir (`$HERMES_HOME/plugins/clawchat`).
- O2. **Hermes ABI surface used**: `ctx.register_platform`, `ctx.register_skill`, `ctx.register_tool`, `ctx.register_cli_command`, `ctx.register_command`, `ctx.register_hook("pre_gateway_dispatch", …)`. None of this is in the docs.
- O3. **Configuration reference**: all `CLAWCHAT_*` env vars (`CLAWCHAT_TOKEN`, `CLAWCHAT_REFRESH_TOKEN`, `CLAWCHAT_BASE_URL`, `CLAWCHAT_WEBSOCKET_URL`/`CLAWCHAT_WS_URL`, `CLAWCHAT_USER_ID`, `CLAWCHAT_AGENT_ID`, `CLAWCHAT_OWNER_USER_ID`, `CLAWCHAT_REPLY_MODE`, `CLAWCHAT_GROUP_MODE`, `CLAWCHAT_GROUP_COMMAND_MODE`, `CLAWCHAT_MEDIA_LOCAL_ROOTS`, `CLAWCHAT_ALLOWED_USERS`, `CLAWCHAT_ALLOW_ALL_USERS`, `CLAWCHAT_HOME_CHANNEL`, `CLAWCHAT_HOME_CHANNEL_NAME`, `CLAWCHAT_HOME_CHANNEL_THREAD_ID`) and the matching `platforms.clawchat.extra` keys + their defaults. Source: `clawchat_gateway/config.py:139-260`.
- O4. **Protocol v2 link**: this plugin and `openclaw-clawchat` are peer Protocol-v2 clients; the source of truth is `clawchat-msghub/docs/features/msghub/protocol-v2-reference.md`. The README never references it.
- O5. **CLI installer entrypoint**: `openclaw-clawchat-cli` (`npx -y @newbase-clawchat/clawchat-cli install --target hermes`) is the canonical user install path; only `SKILL.md:35-37` mentions it.
- O6. **Self-echo hook** (`pre_gateway_dispatch`): unexplained but important.
- O7. **`send_message` target parser patch** for `clawchat:cnv_…` ids: explicit monkey-patch; should be documented as a known integration point.
- O8. **MANIFEST.in does not include `skills/`** even though `__init__.py:236-246` reads `skills/clawchat/SKILL.md`. The Hermes plugin install path copies the whole source tree (`README.md:32`), so this only matters if anyone tries to build a wheel. Flag, don't fix in this pass.
- O9. **No troubleshooting page** for activation failures, missing PATH, container shells.

## 3. Proposed target structure

```
hermes-clawchat/
├── README.md                       # Slim entry-point router (≤80 lines)
├── install.md                      # Operator runbook for AI/LLM use (kept at root, treated as a public artefact)
├── plugin.yaml                     # (unchanged) source of truth for tool list
├── pyproject.toml                  # (unchanged)
├── MANIFEST.in                     # (unchanged)
├── clawchat_cli.py                 # (unchanged)
├── __init__.py                     # (unchanged)
├── clawchat_gateway/               # (code, unchanged)
├── prompts/                        # (unchanged)
├── skills/
│   └── clawchat/SKILL.md           # (unchanged — runtime artefact)
└── docs/
    ├── README.md                   # Audience map ("I want to …")
    ├── install.md                  # Human install + Docker + activation (canonical)
    ├── configuration.md            # Env vars + config.yaml extra reference
    ├── architecture.md             # How the plugin plugs into Hermes (register_platform, hook, patch, skill, tools)
    ├── reference/
    │   ├── tools.md                # 22-tool catalog with one-paragraph descriptions
    │   ├── cli.md                  # /clawchat-activate, hermes clawchat, clawchat_cli.py
    │   └── prompts.md              # platform.md required; user/group optional; defaults
    └── development.md              # uv setup, pytest entry, package layout, .gitignore note
```

Why this shape:

- A root **slim README** for first-touch users (one paragraph + signposts).
- `install.md` stays at the repo root because it is consumed by LLM
  installers (e.g. `openclaw-clawchat-cli`'s prompt at `SKILL.md:30`), and
  external links to that URL must keep working.
- `skills/clawchat/SKILL.md` stays where it is because the Hermes plugin
  loader reads that path at runtime (`__init__.py:238`).
- `docs/` collects everything else, keyed by audience-task.

> **Note on `.gitignore`.** The repo currently ignores `docs/`. We will
> remove that one line so docs get tracked. We will not touch the other
> excluded paths (`tests/`, `.e2e/`, `deploy/`, `scripts/`, `AGENTS.md`,
> `CLAUDE.md`) — Hermes runtime-installs this tree directly into
> `$HERMES_HOME/plugins/clawchat`, and those exclusions exist to keep the
> install payload lean.

## 6. Out of scope (deferred)

- **`MANIFEST.in` does not ship `skills/`** (`O8`). The Hermes runtime install copies the repo tree wholesale (`README.md:32`), so wheel-builds via `setuptools` would lose the skill. Flagging only; this is a packaging fix, not a docs fix.
- **No `tests/` in tree** (gitignored). We do not add tests; the development guide notes only how to set up `uv` + pytest if a `tests/` directory exists locally.
- **No `AGENTS.md` / `CLAUDE.md` added.** Both are gitignored and the plugin is loaded directly into a Hermes install — adding them would either be ignored or would expand the install payload. Out of scope.
- **Protocol-v2 deep contract** is not re-documented here; we only link to the canonical msghub doc.
- **Code edits**: no source edits planned. If a one-line docstring contradicts a verified doc claim during execution, we will flag it in the final report rather than fix it.
- **Sibling-repo claims**: any inconsistency with `openclaw-clawchat/docs/` is out of scope — both plugins are peers of msghub, and the source-of-truth for the wire contract lives in msghub.
